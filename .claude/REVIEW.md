# Code Review Guidelines

## Цель этого документа

Этот документ описывает процесс и критерии code review в проекте.
Цель — единый стандарт качества.

## Перед тем как открывать PR

Автор PR обязан прогнать все проверки локально. Ревьюер вправе отклонить
PR без содержательного ревью, если CI красный.

### Статический анализ
```bash
# Стиль кода (PSR-12 + Slevomat rules)
make phpcs

# Psalm (type-safe анализ)
make psalm

# PHPStan
make phpstan
```

Все три должны завершаться без ошибок.

### Тесты
```bash
make tests
```

Запускается phpunit + infection.
Покрытие не должно падать относительно ветки `main`.
Новая публичная функциональность — обязательно с тестами.
Проверка метрик infection, Mutation Code Coverage должен быть не меньше Covered Code MSI
```shell
Mutation Code Coverage: 100%
Covered Code MSI: 99%
```

## Стиль кода (что проверяет phpcs)

Конфигурация: `phpcs.xml.dist` (PSR-12 + Slevomat Coding Standard).
Ниже — ключевые правила, которые чаще всего вызывают замечания:

**Обязательный `declare(strict_types=1)`**
Каждый PHP-файл начинается с `declare(strict_types=1)` — одна пустая строка
после `<?php`, одна пустая строка после `declare`.

**Trailing comma в массивах**
Многострочные массивы — с trailing comma. Исключение: heredoc.
```php
$config = [
    'foo' => 'bar',
    'baz' => 'qux',  // ← trailing comma обязательна
];
```

**Классы: abstract или final**
Каждый класс должен быть явно `abstract` или `final`.
Нет «просто класса» без модификатора.

**Видимость констант классов**
`const FOO = 'bar'` → только с явным `public`/`protected`/`private`.

**Неиспользуемые импорты**
`use` без использования в коде или аннотациях — ошибка.
Аннотации в docblock тоже учитываются.

**Прочее**
- `self::class` вместо `__CLASS__` / `static::class` где применимо
- Лишние точки с запятой не допускаются
- Дублирующиеся пробелы — ошибка

## Что проверяем в ревью (помимо CI)

### Типизация
- Все параметры и возвращаемые значения типизированы
- Нет `mixed` без обоснования; нет `@psalm-suppress` / `@phpstan-ignore`
  без комментария — *почему* это необходимо

### Архитектура
- Зависимости передаются через конструктор, не создаются внутри методов
- Нет God-объектов: класс решает одну задачу

### Именование
- Имена классов, методов, переменных отражают намерение, а не реализацию
- Аббревиатуры допустимы только если они общеприняты в домене проекта

### Тесты
- Тест проверяет поведение, а не реализацию
- Нет тестов вида «вызвали метод, он не упал»
- Моки — только для внешних зависимостей (IO, HTTP); бизнес-логику тестируем без моков

## Процесс ревью

**Размер PR** — до ~400 строк изменений. Больше — разбивайте.
Исключение: автоматические рефакторинги (переименование, форматирование).

**Время реакции** — ревьюер отвечает в течение одного рабочего дня.
Если нет возможности — предупреждает в комментарии к PR.

**Мёрдж** — только после апрува хотя бы одного ревьюера и зелёного CI.
Автор мёрджит сам.

---

## Частые причины отказа

- CI не прошёл (phpcs / psalm / phpstan / tests)
- Нет типизации в новом коде
- `@psalm-suppress` или `@phpstan-ignore` без объяснения
- Класс без `abstract`/`final`
- Новая функциональность без тестов
- PR больше 400 строк без предупреждения
